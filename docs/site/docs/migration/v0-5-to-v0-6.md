# Migrating from v0.5.x to v0.6.0

v0.6.0 changes the exit-code contract for the `gest` CLI. Before v0.6, almost every
failure path exited with `1` (with one ad-hoc exception: `iteration next` exited with
`2` when no tasks were available). v0.6 replaces that with the BSD
[`sysexits.h`](https://man.openbsd.org/sysexits.3) taxonomy so scripts and agents can
branch on error category via `$?` alone.

No on-disk data migration is required. Project stores, settings, and `.gest/`
layouts from v0.5.x continue to work unchanged on v0.6.

## What changed

Every `cli::Error` variant now maps to a specific sysexits code:

| Code | Name             | Meaning                                         |
|------|------------------|-------------------------------------------------|
| 0    | —                | Success                                         |
| 64   | `EX_USAGE`       | Command-line usage error                        |
| 65   | `EX_DATAERR`     | User data format error                          |
| 66   | `EX_NOINPUT`     | Input entity not found                          |
| 69   | `EX_UNAVAILABLE` | Resource not in the required state              |
| 70   | `EX_SOFTWARE`    | Internal software error (editor failures, etc.) |
| 74   | `EX_IOERR`       | Filesystem or database I/O error                |
| 75   | `EX_TEMPFAIL`    | Try again later (valid empty result)            |
| 78   | `EX_CONFIG`      | Configuration or setup error                    |

See [ADR-0017](https://github.com/aaronmallen/gest/blob/main/docs/design/0017-exit-code-contract.md)
for the full variant-to-code mapping and rationale.

## What you need to update

### Scripts that branch on exit code 1

If you have scripts that check `$? -eq 1` to detect gest errors, update them to
check `$? -ne 0` instead. The non-zero branches now spread across 64–78; a single
bucketed check covers all of them.

```sh
# Before (v0.5.x)
gest task show "$id" || [ $? -eq 1 ] && echo "gest error"

# After (v0.6.0)
gest task show "$id" || echo "gest error"
```

### Scripts that treat `iteration next` exit 2 as "no tasks"

`iteration next` previously exited with `2` when no unblocked tasks were available.
That sentinel is now **`75`** (`EX_TEMPFAIL`). Any orchestrator loop that polls
`iteration next` must update its check:

```sh
# Before (v0.5.x)
gest iteration next "$ITER_ID" --claim --agent worker-1 --json
if [ $? -eq 2 ]; then
  # Advance or idle
fi

# After (v0.6.0)
gest iteration next "$ITER_ID" --claim --agent worker-1 --json
if [ $? -eq 75 ]; then
  # Advance or idle
fi
```

If your script hard-codes the old value `2`, it will now interpret the "no tasks"
signal as a generic error and likely exit the loop.

### Agent profiles that document the old contract

If you wrote local agent profiles (`.agents/profiles/**/SKILL.md` or
`AGENT.md`) that reference "exit code 2 means no tasks available", update them
to match the new contract. The bundled profiles shipped with gest have already
been updated.

## What stayed the same

- `0` still means success.
- Every non-zero code still means "something went wrong or was empty" — the
  `$? -ne 0` idiom continues to work without modification.
- Error messages on stderr are unchanged in wording; only the numeric exit
  code has been retuned.
- No command-line flags or output shapes changed. `--json`, `--quiet`,
  `--raw`, and entity IDs all behave the same.
- No stored data changed. You do **not** need to run any migration command.

## Rolling back

If the new exit codes break something critical, you can downgrade:

```sh
gest self-update --target 0.5.6
```

Your project data remains compatible with v0.5.x.
